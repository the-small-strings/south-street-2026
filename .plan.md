# Walk-on Qobuz Auto-Stop Plan (with Black Buffer Screen)

## Goal
Insert a new plain black screen between `welcome` and `intro` (walk-on), then:
1. Enter black screen
2. Go helper fades Windows master volume down over **2s**
3. Go helper kills Qobuz if running
4. Go helper restores original volume
5. Go helper calls API `next` to auto-start `intro` walk-on animation/music

If helper fails, operator can manually advance from black screen to `intro`.

## Current System Reality (important)
- API currently uses **Socket.IO**, not plain WebSocket (`src/api/src/index.ts`).
- API already emits `gameStateUpdate` whenever state changes.
- Current page flow is `test -> welcome -> intro -> songs...`; we need to change this to include a new black page.
- API is in Docker and exposed to host on `http://localhost:33001`, so a Windows helper app can connect.

## Recommended Architecture

### 1) New page + trigger model (no new socket event)
Add new basic page type (e.g. `walkOnPrep`) rendered as plain black screen.

Updated sequence:
- `welcome -> walkOnPrep -> intro`

Go helper trigger logic:
- Subscribe to existing `gameStateUpdate`
- Track previous and current `currentPage.type`
- Trigger action **only on forward transition** `welcome -> walkOnPrep`
- Ignore reconnect duplicate events and non-forward navigation

This keeps eventing simple and uses existing state updates.

### 2) Windows helper app (Go)
Run a small background app/service on the Windows machine that has Qobuz:
- Connect to API Socket.IO endpoint
- Subscribe to `gameStateUpdate`
- Detect `welcome -> walkOnPrep` (forward only)
- Execute sequence with idempotency lock (prevent duplicate runs)

Execution sequence:
- Read/store current master volume
- Fade down over **2 seconds**
- Kill Qobuz process if present
- Restore exactly to stored volume
- Call `POST /api/game/next` to advance from `walkOnPrep` to `intro`

Failure behavior:
- If any helper step fails, do **not** block the show
- Ensure volume restore is attempted in all failure paths
- Leave system on black screen so operator can press Next manually

### 3) API changes
Minimum required API updates:
1. Add new basic page type (`walkOnPrep`) in shared types.
2. Insert this page in page generation between `welcome` and `intro`.
3. Render black screen in audience UI for `walkOnPrep`.
4. Optionally label in band UI as "Walk-on Prep".

No new socket event is required.

Optional observability endpoint (still useful):
- `POST /api/game/walk-on-completed` for logging helper success/failure telemetry.

### 4) Reliability & safety
- Use one-shot guard per transition (page index + monotonic timestamp).
- Add cooldown window (e.g., 5–10s) to avoid duplicate runs.
- Gate auto-advance: only helper should call `POST /api/game/next` after successful sequence.
- On any failure, always attempt volume restore, then stop and require manual advance.

## Detailed Implementation Steps

### Phase A — Contract
1. Define new page name: `walkOnPrep` (recommended).
2. Define forward trigger contract: `welcome -> walkOnPrep` only.
3. Define helper auto-advance rule: helper calls `POST /api/game/next` only after successful run.

### Phase B — API updates
1. Update API and web type unions to include `walkOnPrep`.
2. Add `walkOnPrep` page in `generatePages` after `welcome` and before `intro`.
3. Ensure `/api/game/next` behavior naturally advances `walkOnPrep -> intro`.
4. (Optional) add `/api/game/walk-on-completed` telemetry route.

### Phase C — Go helper app
1. Build config:
   - `API_URL`, `FADE_DOWN_SECONDS=2`, process names, optional cooldown seconds.
2. Socket.IO client:
   - Connect + auto-reconnect + heartbeat logging.
3. Transition detector:
   - Track previous/current page type and direction.
   - Trigger only on `welcome -> walkOnPrep`.
4. Action runner:
   - Volume control + fade
   - Process termination (`Qobuz.exe`)
   - Restore original volume
5. Auto-advance:
   - Call `POST /api/game/next` after success
   - Retry briefly on transient API failure
6. Packaging:
   - Build as single `.exe`; optional tray icon or Windows service mode.

### Phase D — Test plan
1. Simulate `welcome -> walkOnPrep -> intro` with Qobuz running and not running.
2. Verify exactly-once behavior across reconnects.
3. Verify volume restoration on every failure path.
4. Verify helper auto-advances to `intro` on success.
5. Verify manual advance from `walkOnPrep` works when helper fails/offline.

## Key Flaws / Risks to be aware of
1. **Protocol mismatch risk:** API is Socket.IO; a plain WebSocket Go client will not work.
2. **Missed cue risk:** If helper is offline at `walkOnPrep` transition, auto-run does not happen; manual advance must be used.
3. **System-wide volume side effect:** Master volume fade affects all audio, not just Qobuz.
4. **Process detection variance:** Qobuz process name/window class may vary by install/update.
5. **Security gap:** current API/socket setup is open CORS and unauthenticated; host network clients can connect unless protected.
6. **Race conditions:** repeated advance/back operations can cause duplicate triggers without debouncing/idempotency.
7. **Double-advance risk:** helper `POST /next` can race with user pressing Next; helper should treat 400/invalid-transition as benign and stop.

## Simpler / Alternative Options

### Option 1 (recommended): Black buffer page + existing state updates
Use existing `gameStateUpdate`; helper detects `welcome -> walkOnPrep`, runs sequence, then calls `POST /next`.
- Pros: no new event contract, clean operator fallback
- Cons: requires adding one extra page type

### Option 2: Polling instead of Socket.IO
Helper polls `/api/game/current` every 250–500ms and detects `walkOnPrep` entry.
- Pros: easy Go implementation, no Socket.IO client dependency
- Cons: less elegant, higher request volume, slightly slower response

### Option 3: Dedicated cue endpoint + local automation
When band operator advances to prep screen, trigger a local command/script on Windows (PowerShell/AutoHotkey/StreamDeck), then advance.
- Pros: least distributed complexity
- Cons: tighter coupling to operator workflow; less automatic from API side

### Option 4: Dedicated API event (`walkOnCue`)
API emits explicit cue event when prep begins.
- Pros: clearer semantic contract
- Cons: extra API changes and event lifecycle design

---

## Suggested default path
Start with **Option 1**:
- Add `walkOnPrep` black screen
- Keep existing `gameStateUpdate`
- Implement Windows helper with forward-only transition detection + idempotency
- Helper executes `fade down (2s) -> kill qobuz -> restore original volume -> POST /next`

This gives low implementation risk while still being robust.
